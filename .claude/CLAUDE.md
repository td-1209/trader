# CLAUDE

## ルール

### 作業

- 作業前に`docs/spec/SUMMARY.md`を読み、システムの概要を理解すること
- 実装時は`docs/`の設計書を最新化する
- 設計上の問題や技術的負債を発見した場合、即時ユーザーに報告する
- 自己判断で指示内容や手順を省略・変更しないこと
- 設計書間や設計と実装で矛盾が生じた場合、作業を止め即時ユーザーに報告する
- ユーザーの発言を常に疑い、おかしいと感じた点はその場で指摘すること

### 設計

- 後方互換性を無視し、より技術的負債が少なくなる実装方針とする

### 実装

#### 処理の共通化について

- 1ファイルからのみ呼ばれる → そのファイル内に定義する
- 同フォルダ内の2つ以上のファイルから呼ばれる → そのフォルダ直下に共通ファイルとして配置する
- 異なるフォルダの2つ以上のファイルから呼ばれる → 共通の親階層に配置する
- 共通ファイルが複数ある場合はlibs/などにまとめる
- 呼び出し元が減って1つになった場合、共通ファイルから呼び出し元のファイル内にインライン化する

#### 不要なコードの削除について

- knipを使用して削除
- 関数や変数を基本にexportしない(private)
- 参照が無くなった場合もexportを削除する
- どこからも呼ばれないパッケージ・処理・コンポーネントも削除対象

#### コードテストについて

- AAAパターンで実装する
- テスト名に正しいテスト観点を含ませる(観点: 条件/行動/結果)
- 1テストで1観点だけ実装する
- 境界値・異常系を網羅する
  - 環境差
  - 項目の欠落
  - 不正な入力
  - 存在しないリソース
  - 初回vs既存
- 関数の実装に依存させない(あくまで利用するだけ)
- セットアップ処理はヘルパー関数に隠蔽する(ノイズ除去)
- flakyにしない
  - setTimeoutやsleepなど環境の実行速度に依存する処理を入れない(waitForなどで状態変化を検知)
  - テスト同士を依存させない
    - テスト状態を残留させない
    - 実行順に依存させない
  - 外部サービスに依存させない
    - API
  - その他冪等でない処理に依存させない
    - 日時や乱数
