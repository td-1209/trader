# データ設計

## DBスキーマ

各サービスが自身のPostgreSQLスキーマを所有し、1つのDBインスタンス内で論理分離する。

| スキーマ | サービス | 備考 |
|---|---|---|
| `trading` | 注文執行・取引記録 | 旧record(pl/cf)を統合。`order`はSQL予約語のため`trading`を使用 |
| `research` | 市場調査 | |
| `analysis` | 分析提案 | |

- 各サービスは自スキーマのテーブルのみ直接アクセスする
- 他サービスのデータはHTTP経由で取得する（直接DB参照しない）

## 永続化 vs インメモリ

| データ | 保存先 | 理由 |
|---|---|---|
| ライブ価格ティック | インメモリ | ティックデータはインメモリにし、5分足ごとにOHLCとして1レコードをDBに永続化（trading.candles） |
| オープンポジション | インメモリキャッシュ | `trades.status = 'open'`のレコードをキャッシュ。再起動時はDBから再取得 |
| WS接続状態 | インメモリ | 一時的な状態 |
| 外部APIレート制限カウンタ | インメモリ | 再起動時はリセット（保守的な初期値で安全） |
| Discord通知キュー | インメモリ | Fire-and-forget |
| LLM会話コンテキスト | 非永続化 | 各分析は独立。プロンプトは都度構築 |
| 上記テーブルの全データ | DB | コアビジネスデータ |

## 型定義

### ファイル構成

```
packages/shared/src/schemas/
  trading/
    trades.ts          # 取引用Zodスキーマ（insert/select/update/query）
    cashflows.ts       # キャッシュフロー用Zodスキーマ
    methods.ts         # 手法定義
    candles.ts         # OHLC/ローソク足（WS通信にも使用）
  research/
    news.ts            # ニュース
    calendars.ts       # 経済カレンダー
    sentiment.ts       # センチメント
  analysis/
    analyses.ts        # 分析結果
    strategies.ts      # 戦略提案

services/*/src/db/
  schema.ts            # 各サービスのDrizzleテーブル定義
  migrations/          # Drizzleマイグレーション
```

- Zodスキーマは`packages/shared`に配置する（サービス間通信の型契約）
- Drizzleテーブル定義は各サービス内に配置する（テーブル所有権の明確化）
- 他サービスのテーブルを直接参照するDrizzle定義は作成しない

### DECIMAL型のZod表現

DECIMAL型はJavaScriptの浮動小数点精度の問題を避けるため、Zodでは`z.string()`として扱う。

```typescript
filledPrice: z.string(),    // "1.23456789" — DECIMAL(20,8)
```

### Supabase → Hetzner PostgreSQL 移行手順

1. スキーマ作成（trading, research, analysis）
2. trading.tradesテーブル作成
3. 既存plデータをtrading.tradesに移行（上記カラムマッピングに従う）
4. 既存cfテーブルをtrading.cashflowsに移動（`ALTER TABLE public.cf SET SCHEMA trading; ALTER TABLE trading.cf RENAME TO cashflows`）+カラムリネーム
5. 新規テーブル作成（trading.methods, trading.candles, research.*, analysis.*）
6. methods登録 → tradesのmethodを埋め戻し
7. インデックス作成
8. データエクスポート/インポート（pg_dump → psql）
9. 画像パスの変換（Supabase Storage形式 → ローカルファイルシステム形式）

### updated_atトリガー

既存マイグレーションのトリガー関数を全テーブルに適用する。

```sql
CREATE OR REPLACE FUNCTION trigger_set_updated_at()
RETURNS TRIGGER AS $$
BEGIN
  NEW.updated_at = NOW();
  RETURN NEW;
END;
$$ LANGUAGE plpgsql;
```

updated_atカラムを持つ全テーブルにこのトリガーを設定する。

## 後方互換性による技術的負債

| 問題 | 重要度 | 説明 | 対策 |
|---|---|---|---|
| 旧plデータのカラムマッピング | 高 | entry→entry_price等の変換が必要 | マイグレーションスクリプトで一括変換 |
| 旧plのbase/quote_currency → symbol変換 | 中 | 旧データは分離カラム、新スキーマはsymbol（結合形式） | マイグレーション時に `base_currency \|\| '/' \|\| quote_currency` で変換 |
| methodがフリーテキスト → method(UUID FK) | 中 | 旧plデータはmethod名のテキスト、新スキーマはFK | methods登録後にmethodを埋め戻し |
| reason/result_imageのパス形式混在 | 低 | Supabase形式と新形式が混在する | アプリ層で両形式対応、後日一括変換 |

## 共通規約

- タイムスタンプ: 全てTIMESTAMPTZ（UTC保存）。表示時のみJST変換
- PK: 全テーブルUUID v4（gen_random_uuid()）。serial/auto-incrementは使用しない
- ソフトデリート: 不使用。取引履歴は不変データとして扱う。手法・戦略はstatusフラグで管理
- JSONB: 拡張性のある半構造化データにのみ使用（手法設定、分析メタデータ）。頻繁にクエリするデータには使用しない
- 配列型: TEXT[]・UUID[]は限定的に使用。クエリ対象の場合はGINインデックスを付与
