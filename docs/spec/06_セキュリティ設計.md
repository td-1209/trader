# セキュリティ設計

## 認証・認可

### Basic認証

Caddyで全エンドポイントにHTTP Basic認証を適用する。個人利用のため、セッション管理やJWT等の複雑な認証は不要。

| 項目 | 内容 |
|---|---|
| 方式 | HTTP Basic Authentication |
| 適用範囲 | Caddy経由の全リクエスト（API + Web UI） |
| ユーザー数 | 1人（自分のみ） |
| 認証情報 | 環境変数で管理（`BASIC_AUTH_USER`, `BASIC_AUTH_PASSWORD`） |

### サービス間通信

Docker Composeの内部ネットワークで完結するため、サービス間の認証は不要。外部からはアクセスできない。

```
インターネット → Caddy（Basic認証） → 各サービス
                                    ↕ サービス間（認証なし・内部ネットワーク）
```

## ネットワーク

### 外部公開

Caddyのみが外部ポート（80/443）を公開する。他のサービス・DBは外部からアクセスできない。

```
外部公開: Caddy（80/443）
内部のみ: web(3000), trading(3001), research(3002), analysis(3003), postgres(5432)
```

### ファイアウォール

UFWで以下のみ許可（cloud-initで自動設定）：

| ポート | 用途 |
|---|---|
| 22 | SSH |
| 80 | HTTP（→ 443にリダイレクト） |
| 443 | HTTPS |

### HTTPS

CaddyがLet's Encrypt証明書を自動取得・更新する。HTTPは全て443にリダイレクト。

## APIキー管理

全てのAPIキー・認証情報は環境変数で管理する。

- `.env`ファイルに記載し、`.gitignore`でGit管理対象外とする
- ソースコードにハードコードしない
- CI/CDではGitHub Secretsに格納する

環境変数の具体的な一覧は運用デプロイ設計を参照。

## バリデーション

### 入力バリデーション

全てのAPIリクエストをZodスキーマで検証する。スキーマは`packages/shared`に定義し、サービス間で共有する。

| 対象 | 方式 |
|---|---|
| リクエストボディ | Zodスキーマでparse |
| クエリパラメータ | Zodスキーマでparse |
| パスパラメータ | UUID形式のバリデーション |

バリデーション失敗時は400レスポンスにフィールド単位のエラーを返す（詳細はAPI設計を参照）。

## CORS

単一ドメイン（`trade.example.com`）でCaddyがパスベースルーティングを行うため、本番環境でのCORS設定は不要。全リクエストが同一オリジンとなる。

ローカル開発時（`localhost:3000` → `localhost:3001`等）はオリジンが異なるため、開発環境のみCORSヘッダーを設定する。
