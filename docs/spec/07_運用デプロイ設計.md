| 項目 | 内容 |
|---|---|
| プラン | CX32（4GB RAM / 2 vCPU / 40GB SSD） |
| リージョン | シンガポール（日本からのレイテンシ約60〜80ms） |
| 帯域 | 20TB/月 |
| 月額 | €7.45（約1,200円） |

### Windows VPS（MT5稼働用）

ABLENET VPS Win1プランでMT5を常時稼働させる。

| 項目 | 内容 |
|---|---|
| プロバイダー | ABLENET VPS |
| プラン | Win1（2GB RAM / 仮想2Core / 60GB SSD） |
| OS | Windows Server 2022 |
| リージョン | 大阪 |
| 用途 | MT5常時起動 + ZeroMQブリッジEA |
| 接続 | Linux VPSからZeroMQで通信 |
| 月額 | 約1,587円〜（新規価格） |

### Docker Compose構成

```yaml
services:
  caddy:
    # リバースプロキシ（ポート80/443公開）
  web:
    # Next.js（:3000）
  trading:
    # Hono（:3001）
  research:
    # Hono（:3002）
  analysis:
    # Hono（:3003）
  postgres:
    # PostgreSQL（:5432、内部のみ）
```

### Caddyルーティング

```
trade.example.com/api/trading/*  → trading:3001
trade.example.com/api/research/* → research:3002
trade.example.com/api/analysis/* → analysis:3003
trade.example.com/*              → web:3000
```

Caddyが自動でLet's Encrypt証明書を取得・更新する。

### XMTrading（注文執行）

XMTradingは独自のREST APIを提供していない。MT5（MetaTrader 5）経由でのみプログラマティックアクセスが可能。Windows VPS上でMT5を常時起動し、EA（Expert Advisor）がtradingサービスのHTTPエンドポイントをポーリングして注文を中継する。

```
MT5 EA (Windows VPS) →HTTP polling→ trading service (Linux VPS)
MT5 EA (Windows VPS) →HTTP POST→   trading service (Linux VPS)  [結果返送]
MT5 EA (Windows VPS) → XMTrading  [実際の注文執行]
```

| 項目 | 内容 |
|---|---|
| 利用サービス | trading |
| 実行環境 | Windows VPS上のMT5（常時起動） |
| ブリッジ方式 | HTTP polling（MT5 EA → trading service） |
| EA言語 | MQL5 |
| 機能 | 注文発注・決済・ポジション同期・約定結果取得 |
| 認証 | Basic Auth（Caddy経由）+ XMTrading MT5口座情報 |

## IaC

### Terraform

VPSの作成・ファイアウォール・SSHキーをコードで管理する。

```
infra/
├── main.tf           # Hetznerリソース定義
├── variables.tf      # 変数定義
├── outputs.tf        # IPアドレス等の出力
├── cloud-init.yml    # サーバー初期設定
└── terraform.tfvars  # 環境固有値（.gitignore対象）
```

`terraform apply` でVPS作成 → cloud-initが自動実行される。

### cloud-init

VPS初回起動時に以下を自動設定：

- Docker + Docker Compose インストール
- 非rootユーザー作成
- SSH鍵設定・パスワード認証無効化
- ファイアウォール（ufw: SSH/HTTP/HTTPSのみ許可）

## デプロイフロー

```
terraform apply → VPS作成・cloud-init実行
                → SSH接続可能に
                → git clone → docker compose up -d
```

以後のデプロイはCI/CDで自動化。

## CI/CD

### GitHub Actions

mainブランチへのpushで自動デプロイする。

```
push to main → GitHub Actions → SSH into VPS → git pull → docker compose up -d --build
```

1. VPSにSSH接続（秘密鍵はGitHub Secretsに格納）
2. `git pull origin main`
3. `docker compose up -d --build`（更新されたサービスのみリビルド・再起動）
4. ヘルスチェック（各サービスの `/health` にcurl）
5. 失敗時はDiscord Webhookで通知

### GitHub Secrets

| Secret | 用途 |
|---|---|
| `VPS_HOST` | VPSのIPアドレス |
| `VPS_SSH_KEY` | SSH秘密鍵 |
| `VPS_USER` | SSHユーザー名 |

## バックアップ

### データベース

- `pg_dump`で日次バックアップ
- 保存先: VPSローカル + Hetzner Object Storage（S3互換）
- 保持期間: 30日
- cronで自動実行

### 画像ファイル

- 保存先: VPSローカル（`/data/images/trades/{id}/`）
- Hetzner Object Storage（S3互換）へcronでrsync
- 保持期間: 無期限

## 監視

- 各サービスに`/health`エンドポイントを設置
- cronで定期的にヘルスチェック
- 異常検知時はDiscord Webhookで通知
- Docker Composeのrestart policyで自動再起動（`restart: unless-stopped`）

## ログ

- 各サービスはstdout/stderrに出力
- Docker Composeのデフォルトログドライバーで保持
- `docker compose logs -f <service>` で確認
- ログローテーション: Dockerデフォルト設定（max-size: 10m, max-file: 3）

## 障害対応

### サービス停止時

1. `docker compose ps` でサービス状態を確認
2. `docker compose logs <service>` でエラー内容を確認
3. `docker compose restart <service>` で再起動
4. 改善しない場合は `docker compose up -d --build <service>` でリビルド

### VPS障害時

1. Hetzner Cloudコンソールから状態を確認
2. 復旧不能な場合は `terraform apply` で新VPSを作成
3. Hetzner Object Storageからバックアップをリストア
4. `docker compose up -d` で全サービスを起動

### ロールバック

1. `git revert` or `git checkout <commit>` で対象コミットに戻す
2. `docker compose up -d --build` でリビルド・再起動
